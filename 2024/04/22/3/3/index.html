以及Springboot特有的报错页面

shiro简介

Apache Shiro是一个Java 的安全(权限)框架

shiro框架提供了记住密码的功能，用户登录成功后会生成经过加密并编码的cookie给用户。服务端会将cookie先base64解码，再AES解密，再反序列化。这里就包含了反序列化RCE漏洞，如果我们反向以上操作，将命令=>序列化=>AES加密=>base64加密，放在rememberMe处，就可以触发。

这个过程中最难的是AES加密部分，因为AES加密是需要密钥的，而我们不知道服务端的密钥是多少。

通过代码阅读可以知道，源代码没有特别设定密钥key，因此在这个版本的shiro会随机生成密钥。

Shiro-550

这个漏洞跟本题无关，但也值得讲一下。

影响：Apache Shiro < 1.2.4

这里只展示关键部分

如上，在1.2.4以前，Apache Shiro的key默认密钥是固定的，因此可以轻松进行以上描述的反序列化RCE。

heapdump泄露

那么我们如何获取key呢？Springboot有一种信息泄露是heapdump泄露。

在资源文件/BOOT-INF/classes/application.properties中可以看到

这句management.endpoints.web.exposure.include=*

Spring Boot Actuator提供了很多管理端点，用于监控和管理Spring Boot。这句话的意思是暴露所有的端点。

黑盒时可以直接尝试访问一下

成功了，访问/actuator/heapdump将heapdump下载下来。heapdump可以用各种工具查看。

比如我使用JDumpSpider，java -jar .\JDumpSpider-1.1-SNAPSHOT-full.jar .\heapdump > 1.txt

然后可以直接在1.txt中检索到key

获取key了，接下来的流程跟shiro-550几乎一样了（除了新版本加密方式换了以外）

这里偷懒使用github上找的工具shiro_attack2

直接输入key就可以了。shiro在1.4.2之后更换了加密方式（与shiro-721漏洞相关），即图中被勾选上的AES GCM。
